# Objective 26.5 — App Chrome (Header / Footer / Side Nav / Breadcrumb)

**Phase:** 1 (AI Pipeline Module) · **Module:** UI Generation
**Depends on:** Obj 26 (Stage 6 UI Generation), Obj 15.5 (Workspace Assets), Blocks Library system
**Feeds into:** Obj 27 (Stage 7 Code Generation), Obj 30 (Stage 10 Maintenance)

---

## Why this exists

Generated apps have surfaces that *every page* shares — the top header, the side navigation, the footer, the breadcrumb. These are not page-level concerns and should not be duplicated in every page's layout.

The original 30 objectives implied this in passing ("generated apps look real") but didn't define it as a concrete subsystem. Without explicit chrome management:
- Each page would have to redeclare its own header/footer
- Site-wide nav changes would require editing N pages
- The AI pipeline would have to regenerate chrome on every page generation

App Chrome makes the header / footer / side nav / breadcrumb a **first-class app-level concern**, configured once and applied to every page (with per-page overrides where needed).

---

## What this provides

### App Chrome surface

A dedicated UI in the platform (one per generated app) that lets the user:

1. **Choose a layout** — sidenav-with-topbar OR topnav-only
2. **Pick a chrome block for each region** — header, side nav, breadcrumb, footer
3. **Configure region-specific parameters** — product name, search visibility, breadcrumb separator, footer link groups, etc.
4. **Define page-level overrides** — e.g., `/sign-in` renders without sidenav; `/embed/*` renders without any chrome.

### Chrome blocks as a Blocks Library category

Chrome regions are filled by **chrome blocks**: a special category in the Blocks Library marked with category=Chrome. These blocks declare which region they're for via a `chrome` field:

```yaml
id: chrome-header-app
kind: tailwind
category: Chrome
chrome: header  # 'header' | 'footer' | 'sidenav' | 'breadcrumb'
params:
  - logoUrl
  - productName
  - navItems
  - showSearch
  - userMenuItems
```

This means:
- Anyone can build a custom chrome block in the Blocks Library and it will appear in the App Chrome region picker.
- Chrome blocks are versioned, share-able across projects in the same workspace, and follow the same registry workflow as any other block.
- The platform ships a starter set: `chrome-header-app`, `chrome-header-marketing`, `chrome-header-minimal`, `chrome-sidenav-vertical`, `chrome-sidenav-icon-only`, `chrome-breadcrumb`, `chrome-breadcrumb-tabbed`, `chrome-footer-standard`, `chrome-footer-minimal`. Each can be replaced.

### How chrome renders in the generated app

The generated app's root layout component receives the chrome configuration and renders each region around the page content. The Designer canvas mirrors this exactly — chrome regions show on every page with a "Configured in App Chrome ↗" lock indicator, so editors can see context but can't accidentally edit the chrome from a page view.

```jsx
// Simplified generated layout
<AppShell>
  <ChromeRegion region="header" override={pageOverrides.header} />
  {layout === 'sidenav' && (
    <ChromeRegion region="sidenav" override={pageOverrides.sidenav} />
  )}
  <main>
    <ChromeRegion region="breadcrumb" override={pageOverrides.breadcrumb} />
    {pageContent}
  </main>
  <ChromeRegion region="footer" override={pageOverrides.footer} />
</AppShell>
```

### Page-level overrides

Pages can declare chrome overrides in their metadata:

```yaml
# pages/sign-in.page.yaml
chromeOverride:
  layout: topnav-only       # no sidenav for this page
  sidenav: none
  breadcrumb: none
```

```yaml
# pages/embed/widget.page.yaml
chromeOverride:
  layout: full-page          # no chrome at all
  header: none
  sidenav: none
  breadcrumb: none
  footer: none
```

The Designer surfaces these overrides in the page metadata inspector. AI can suggest overrides ("This is a sign-in page — most apps render it full-page. Apply override?").

### AI Pipeline integration

During Stage 6 (UI Generation), the AI does NOT generate chrome per-page. Instead:

1. On the first run for a new project, Stage 6 also produces an **App Chrome proposal** — picks chrome blocks from the workspace library that match the brand and PRD.
2. The user reviews and approves the chrome proposal alongside the first set of generated pages.
3. Subsequent page generations consume the approved chrome and render content within it.

If the user later changes the App Chrome configuration, all pages re-render with the new chrome (no regeneration needed — just a re-build).

---

## Acceptance criteria

1. App Chrome surface lets the user configure header, side nav, breadcrumb, footer for an entire app.
2. The Blocks Library supports a Chrome category; chrome blocks declare their target region via a `chrome` field.
3. The platform ships at least 2 chrome blocks per region as a starter set.
4. Page metadata supports chrome overrides at the layout level (sidenav-only / topnav-only / full-page) and per-region (none / specific block id).
5. Designer canvas renders chrome regions around the page content with a visible "Configured in App Chrome" lock indicator.
6. AI Pipeline Stage 6 generates an initial App Chrome proposal alongside the first page batch; user can review and approve.
7. Changing App Chrome configuration triggers a rebuild but does NOT require regenerating pages.

---

## Out of scope

- Per-route chrome (different chrome for different sections of the app) beyond the simple page-level override — chrome is app-wide.
- Multiple chrome variants A/B-tested. Chrome variations should be handled via separate environments or branches.
- Chrome animation choreography (collapse-on-scroll, hide-on-scroll-down) — chrome blocks may implement these themselves; not a platform concern.
