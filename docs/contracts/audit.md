# Contract: Audit

> Port: `@platform/ports-audit` — `AuditPort`

## Purpose

`AuditPort` provides an append-only audit log for the platform. Every significant actor-driven or system-driven operation writes an entry describing who did what to which resource and when. The audit log is the authoritative record for compliance, debugging, and support.

Two invariants are non-negotiable:

1. **Append-only.** There is no update or delete method on this port. Audit entries, once written, are immutable. Any adapter implementation that permits modification of audit entries is non-conformant.
2. **Out-of-band write.** `write()` must be called after the operation succeeds and must use a separate database connection or session — never inside the same transaction as the operation being audited. This guarantees that the audit entry survives even if the operation's transaction is later rolled back by a subsequent error.

## Types

```typescript
interface AuditEntry {
  id?: string; // Generated by the adapter if omitted
  workspaceId: string;
  actorId: string | null; // null for system-generated events
  action: string; // Follows 'resource.verb' convention
  resourceType: string;
  resourceId: string;
  before?: Record<string, unknown>; // State before the operation (for updates)
  after?: Record<string, unknown>; // State after the operation (for updates/creates)
  metadata?: Record<string, unknown>; // Additional context (e.g., inviteCode, source IP)
  occurredAt: Date;
  ipAddress?: string;
  userAgent?: string;
}

interface AuditFilter {
  workspaceId?: string;
  actorId?: string;
  action?: string;
  resourceType?: string;
  resourceId?: string;
  from?: Date;
  to?: Date;
}

interface Page {
  limit: number;
  offset: number;
}

interface PaginatedResult<T> {
  items: T[];
  total: number;
}

type AuditErrorCode = 'WRITE_FAILED' | 'QUERY_FAILED' | 'UNKNOWN';
```

## Methods

### `write(entry): Promise<Result<void, AuditError>>`

Appends a single audit entry to the log.

**Pre-conditions:**

- `entry.workspaceId` is non-empty.
- `entry.action` follows the `resource.verb` naming convention (e.g., `'schema.created'`, `'user.invited'`, `'artifact.approved'`). Actions outside this convention are accepted but will not appear in standard audit UI filters.
- `entry.actorId` is null for events triggered by background tasks, scheduled jobs, or system processes (not a user).
- `entry.occurredAt` should reflect when the operation occurred, not when the audit entry is written. These are usually the same, but in replay or migration scenarios they may differ.
- `entry.before` and `entry.after` should contain only the relevant fields that changed, not the entire resource. Avoid including large blobs or binary data.

**Post-conditions:**

- On success, returns `ok(undefined)`. The entry is durably persisted (for production adapters) or stored in memory (for the in-memory adapter).
- The adapter assigns `entry.id` if it was not provided; the assigned ID is not returned (the caller does not need it).
- On failure, returns `err({ code: 'WRITE_FAILED', ... })`. The caller should log the failure but must not block the operation that triggered the write — audit write failures are non-fatal.
- Under no circumstances does `write()` modify the operation that preceded it. It is a side effect, not a step in the operation.

**Calling convention:**

```typescript
// Correct: write AFTER the operation succeeds, outside its transaction
const createResult = await this.repo.create(input); // commits its own transaction
if (createResult.isErr()) return err(createResult.error);

await this.audit.write({
  workspaceId: ctx.workspaceId,
  actorId: ctx.actorId,
  action: 'schema.created',
  resourceType: 'schema',
  resourceId: createResult.value.id,
  after: { name: createResult.value.name },
  occurredAt: new Date(),
});

return ok(createResult.value);
```

---

### `query(filter, page): Promise<Result<PaginatedResult<AuditEntry>, AuditError>>`

Returns a paginated, filtered view of the audit log.

**Pre-conditions:**

- At least one filter field should be set in production use. An entirely empty filter (all fields undefined) returns the full audit log across all workspaces — this is permitted for installation-level admin queries only. Service-level callers must always set `workspaceId`.
- `page.limit` must be between 1 and 1000. Values outside this range are clamped or rejected by the adapter.
- `filter.from` and `filter.to` are inclusive.

**Post-conditions:**

- Returns `ok({ items, total })` where `total` is the count of entries matching the filter before pagination.
- `items` are ordered by `occurredAt` descending (most recent first) unless the adapter specifies otherwise.
- String filters (`action`, `resourceType`, `actorId`, etc.) are exact matches, not prefix or substring searches. Partial filtering (e.g., all actions on a resource type) is done by omitting more-specific filters.
- Returns `ok({ items: [], total: 0 })` when no entries match — never returns `ok(null)`.
- On backend failure, returns `err({ code: 'QUERY_FAILED', ... })`.

## Capability Flags

| Flag                   | Meaning                                                                                    |
| ---------------------- | ------------------------------------------------------------------------------------------ |
| `audit.fullTextSearch` | Adapter supports substring/keyword search within `action` and `metadata` fields            |
| `audit.retention`      | Adapter enforces a configurable retention period; entries older than the period are purged |
| `audit.exportCsv`      | Installation-level admin can export filtered audit logs as CSV                             |

## Performance Expectations

| Operation                           | Target latency                                                                          |
| ----------------------------------- | --------------------------------------------------------------------------------------- |
| `write` (production adapter)        | < 20 ms p99; write path is fire-and-forget from the caller's perspective                |
| `query` (limit 50, indexed fields)  | < 100 ms p99                                                                            |
| `query` (limit 50, date range only) | < 500 ms p99 (may require full scan on large tables without per-workspace partitioning) |

Production adapters should index `(workspaceId, occurredAt)` at minimum. `(workspaceId, resourceType, resourceId)` is recommended for resource-specific lookups. Avoid queries that filter by `actorId` alone without also filtering by `workspaceId`.

## Known Adapter Divergences

| Behaviour             | In-memory                              | Production (Postgres/MSSQL/MongoDB)                                     |
| --------------------- | -------------------------------------- | ----------------------------------------------------------------------- |
| Durability            | None — entries lost on process restart | Persisted to `audit_log` table or collection                            |
| `id` assignment       | UUID generated in-memory               | Database-generated (serial or UUID)                                     |
| Retention policy      | None                                   | Configurable; enforced by background job                                |
| `query` ordering      | Insertion order                        | `occurredAt` descending                                                 |
| Cross-workspace query | Permitted (no enforcement)             | Permitted for admin context only; enforced by RBAC layer above the port |
| Write isolation       | Single-process only                    | Separate connection from operation transaction                          |

The in-memory adapter is the correct choice for unit tests. Use it to verify that `write()` is called with the right arguments using a spy — do not test durability or query indexing at the unit level.

## Usage Examples

```typescript
// Logging a user invitation
await this.audit.write({
  workspaceId: ctx.workspaceId,
  actorId: ctx.actorId,
  action: 'user.invited',
  resourceType: 'user',
  resourceId: newUser.id,
  after: { email: newUser.email, role: assignment.role },
  metadata: { inviteCode: token },
  ipAddress: ctx.ipAddress,
  userAgent: ctx.userAgent,
  occurredAt: new Date(),
});

// Logging a system-initiated artifact approval
await this.audit.write({
  workspaceId: job.payload.workspaceId,
  actorId: null, // system actor
  action: 'artifact.approved',
  resourceType: 'artifact',
  resourceId: artifact.id,
  before: { status: 'pending_review' },
  after: { status: 'approved' },
  metadata: { triggeredBy: 'scheduled-review-job' },
  occurredAt: new Date(),
});

// Querying the audit log for a specific resource
const result = await this.audit.query(
  {
    workspaceId: ctx.workspaceId,
    resourceType: 'schema',
    resourceId: schemaId,
  },
  { limit: 25, offset: 0 },
);

if (result.isOk()) {
  const { items, total } = result.value;
}
```

## Common Misuse

**Writing the audit entry inside the operation's transaction.** If the transaction is rolled back for any reason after `write()`, the audit entry is rolled back with it — no record of the attempted operation survives. Always write audit entries after the transaction commits, using a separate connection or session.

**Writing the audit entry before the operation succeeds.** An audit entry for an operation that never completed is misleading and potentially dangerous (false compliance record). Check the operation result first; only write the audit entry on success.

**Making audit write failures fatal.** A failing audit write should be logged at error level but should not fail the operation that triggered it. The operation has already succeeded; the audit entry is a side effect. Return the operation's result to the caller regardless.

**Including sensitive data in `before`/`after`.** Do not store passwords, API keys, access tokens, or PII in audit entry fields. Store references (IDs) and non-sensitive descriptors. If a field is redacted in the API response, it must also be redacted in the audit entry.

**Using audit entries as an event bus.** The audit log is a compliance and debugging record, not an event queue. Do not poll the audit log to trigger downstream behaviour. Use `EventBusPort` (see `eventing.md`) for event-driven workflows.

**Omitting `workspaceId` on service-level writes.** Audit entries without a `workspaceId` are invisible to workspace-scoped audit queries. Every service-level operation has a workspace context; always supply it.

**Using free-form action strings.** Actions must follow `resource.verb` convention and should be drawn from the platform's audit vocabulary (defined in Objective 8). Inventing ad-hoc strings makes audit UI filters and compliance reports unreliable.
