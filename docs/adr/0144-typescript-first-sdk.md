# ADR-0144: TypeScript-First SDK

**Status:** Accepted
**Date:** 2026-05-06
**Objective:** 19 (Public SDK)

## Context

The platform needs a client SDK that customer developers use to build their applications. The platform targets three kinds of customers: JavaScript/TypeScript frontend developers, Node.js backend developers, and polyglot teams that may use Python or Go. A decision is needed on which language to implement first and how to structure the multi-language story.

## Decision

TypeScript is the source language for the SDK. The package `@platform/sdk` is authored in TypeScript and compiled to dual ESM + CJS output. All other language SDKs (Python, Go, Rust, Swift, Kotlin) are deferred until customer demand justifies them.

TypeScript is chosen because:

- The platform's primary audience in v1 is JavaScript/TypeScript developers (the same demographic as Supabase)
- TypeScript covers both browser and Node.js runtimes from a single codebase
- OpenAPI 3.1 specs (already generated by Objective 12) and GraphQL schemas enable straightforward language-agnostic type generation for future SDKs
- The JS ecosystem's tooling (tsup, TanStack Query, Vitest) matures faster than equivalents in other languages

The compiled output (ESM + CJS `.js` + `.d.ts`) works for JavaScript users who don't use TypeScript — they get autocomplete and documentation via the `.d.ts` types without needing the TypeScript compiler.

## Consequences

**Easier:**

- Single codebase covers browser, Node.js, Deno, Bun, React Native, Cloudflare Workers
- React and Vue integration packages can be written in the same language with shared types
- Type generation from OpenAPI/GraphQL is first-class

**Harder:**

- Python and Go customers must wait for future SDK objectives
- The SDK cannot use Node.js-only APIs in the universal build without runtime branching

**Alternatives considered:**

- **Universal JavaScript (no types):** Faster initial development; rejected because types are the primary DX differentiator vs. Supabase
- **Python first:** Rejected — smaller immediate market; TypeScript covers more of v1 use cases
- **Multi-language from day one:** Rejected — spreads implementation effort; OpenAPI specs make future languages tractable when demand arrives
