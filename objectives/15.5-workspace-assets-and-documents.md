# Objective 15.5 — Workspace Assets and Documents

**Phase:** 0 (Foundation) · **Module:** Data Management Module
**Depends on:** Obj 15 (Storage Browser & File Management), Obj 06 (Multi-tenancy & RBAC)
**Feeds into:** Obj 22 (Stage 2 PRD), Obj 23 (Stage 3 Design Tokens), Obj 26 (Stage 6 UI Generation), Obj 27 (Stage 7 Code Generation)

---

## Why this exists

The original 30 objectives covered project-level storage (Obj 15) but not the workspace-level concept of **brand assets and reference documents** that the AI Pipeline needs as context.

Without this layer, every project would have to re-upload logos, restate brand voice, and re-explain the company's existing material. With it, the AI Pipeline always knows what the workspace looks and sounds like — and reuses that context across every project.

This is the difference between an AI that generates _generic-looking output_ and an AI that generates _output that fits the company_.

---

## What this provides

### Asset categories

**Brand assets** (visual identity, used by Stage 3 design token generation):

- Logos — primary, mark-only, monochrome dark, monochrome light. Stored as SVG when possible; raster fallbacks generated.
- Brand colors — declared in OKLCH (canonical) with hex / RGB shown alongside. Each color tagged with usage role (primary, accent, success, warning, danger, surface, etc.).
- Fonts — registered font families with weight inventory, web-format optimised (woff2, variable when available). Self-hosted by default; CDN registration optional.
- Images — hero shots, illustrations, photography. Tagged with usage hint ("marketing", "team", "feature illustration").
- Icons — custom icon sets beyond the built-in icon library.

**Reference documents** (textual context, used by Stage 2 PRD and Stage 7 code generation):

- Brand voice guidelines — tone, vocabulary, do's and don'ts.
- Strategy documents — product strategy, vision, positioning.
- Existing material — screenshots of legacy systems being replaced, sample customer communication.
- Compliance requirements — GDPR addenda, internal data handling rules, regulatory constraints.
- Technical specs — API specs of systems being integrated with, OAuth scope requirements.

### Storage and access

Assets live in the workspace storage volume, separate from project-level storage. Path convention:

```
/workspaces/{workspace_id}/assets/
  brand/
    logos/
    colors/
    fonts/
    images/
    icons/
  documents/
    voice/
    strategy/
    reference/
    compliance/
    specs/
```

RBAC: workspace admins can write; all members can read. Documents added here are NOT exposed via the project-level public API — they're internal context only.

### AI Pipeline integration

Each pipeline stage that benefits from workspace context is configured to read from specific asset categories. The configuration is declarative (an objective in code) so it can be audited and changed:

```typescript
// Stage 2 (PRD) reads from:
const stage2Context = {
  documents: ['voice', 'strategy', 'reference'],
  format: 'plain_text_concatenated_with_separators',
};

// Stage 3 (Design Tokens) reads from:
const stage3Context = {
  brand: ['logos', 'colors', 'fonts'],
  documents: ['voice'], // for typography decisions
};

// Stage 6 (UI Generation) reads from:
const stage6Context = {
  brand: ['logos', 'colors', 'fonts', 'images', 'icons'],
  documents: ['voice', 'reference'],
};

// Stage 7 (Code Generation) reads from:
const stage7Context = {
  documents: ['compliance', 'specs'],
};
```

Reference documents are added to the AI's context window with provenance tags so generated artifacts can cite which document they drew from.

### Telemetry and provenance

Every generation that consumed workspace assets records _which assets influenced it_. This appears in the Generation History audit trail. If the brand voice document is updated, the platform can identify which previous generations would now produce different output and flag them for review.

---

## Acceptance criteria

1. A workspace admin can upload, replace, and delete brand assets and reference documents.
2. Each pipeline stage reads only from the asset categories it's declared to read from (least-context principle).
3. Asset uploads trigger re-validation: oversized images flagged, fonts checked for web-safe formats, colors checked for WCAG AA compliance.
4. The AI Pipeline's Generation History records which assets each generation referenced.
5. Updating a document does NOT auto-regenerate prior outputs — it flags them for review with a "would change if regenerated" indicator.
6. Storage usage is tracked against the workspace allowance (1 GB default; configurable per plan).
7. Standard image formats (SVG, PNG, JPG, WebP, AVIF) and document formats (MD, PDF, DOCX, TXT) are supported. Other formats stored but not parsed for context.

---

## Out of scope

- Asset _generation_ (e.g., AI logo creation) — assets are uploaded by humans.
- DAM-style metadata (rights management, expiration dates, usage rights) — captured as document text if needed but not first-class.
- Live asset CDN distribution — that's the project storage layer's job.
