# Objective 9.5: Platform Upgrade & Versioning

**Status:** Ready for development
**Prerequisites:** Objectives 1–9 (foundation); Obj 4/4a/4b shipped the `platform_versions` table backfill (migrations 0009 / 0007 / 0007 respectively).
**Blocks:** Customer self-host adoption beyond v1.0 — without this, customers cannot safely move from v1.0 to v1.1.

**Source:** Promoted from V2 Layer E (E1) to V1 on 2026-05-05 because "self-hostable" is a thesis pillar; without an upgrade path it collapses on the second release.

---

## 1. Purpose

Make the self-hosted platform **safely upgradable** across versions. A customer running v1.0 must be able to move to v1.1, then v1.5, then v2.0 — across PostgreSQL, MSSQL, or MongoDB — with documented procedures, predictable downtime, working rollback, and HA-aware coordination.

Today the platform has migration discipline (Obj 4: per-DB migration runners with checksums and journals) and a Windows blue-green runbook (Obj 9). What it _doesn't_ have:

- A platform-release-version concept (`platform_versions` table now exists per Obj 4 backfill but nothing reads or writes it)
- A coordinated upgrade orchestrator that runs migrations + version recording + health checks across all three databases
- Cross-version compatibility window declaration (can a v1.0 customer skip directly to v1.5?)
- Version-skew handling for HA deployments (multiple platform instances during a rolling upgrade)
- A documented upgrade runbook customers actually follow

This objective builds those pieces. The output is a platform that customers can run in production for years without being stranded on an old version.

---

## 2. Scope

### In Scope

- **Platform version concept**: a single source of truth for "what version is this platform" (read from `package.json` + git tag), exposed via a `PlatformVersionPort` and a CLI
- **Version recording**: every successful upgrade inserts a row into `platform_versions` on every active database
- **Upgrade orchestrator** (`platform upgrade`): one command that runs schema migrations + records the version + verifies health across all configured databases
- **Compatibility window policy**: declared in code; the orchestrator refuses upgrades that violate it
- **Rollback support**: the orchestrator can roll back to the previous version via the existing `down` migration paths plus a corresponding `platform_versions` row removal
- **HA / version-skew handling**: rolling upgrade procedure (one instance at a time) with a documented compatibility contract — N and N-1 platform versions must coexist for the duration of a rolling restart
- **Customer-facing upgrade runbook**: step-by-step, with expected downtime and rollback steps
- **Operator UI panel** (in installation admin): current version, last upgrade, eligible upgrades, "upgrade now" with confirmations
- **Pre-flight checks**: backup verification, free disk, replication lag, active long transactions
- **Telemetry**: every upgrade emits structured events (start, per-step, complete, failure) for observability
- **CI verification**: an end-to-end test that boots a platform on v(n-1), runs the orchestrator to v(n), and verifies state on all three databases

### Out of Scope (Belongs to Later Objectives)

- **Major-version migrations that require data reshape** (v1 → v2 if it ever happens): handled per-major-version via a dedicated migration objective at the time
- **Customer data export/import** during an upgrade: handled by Obj 7 (compliance / data subject rights) and the standard backup mechanism
- **Customer-defined schema migrations** (their data, not platform data): handled by Stage 4 (Schema Synthesis, Obj 24) and the Data Management Module
- **Zero-downtime upgrades for breaking schema changes**: the policy is "minimal downtime; breaking changes get scheduled maintenance windows"

## 3. Locked Decisions

| Decision                             | Choice                                                                                                        | Rationale                                                                     |
| ------------------------------------ | ------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------- |
| Source of truth for platform version | Root `package.json` `version` field, mirrored at build time into `packages/core/src/platform/version.ts`      | Single canonical place; no drift between docs and code                        |
| Version format                       | Semver (`MAJOR.MINOR.PATCH[-prerelease]`)                                                                     | Industry standard; tooling-friendly                                           |
| `platform_versions` is per-database  | Yes — recorded on every active DB on every upgrade                                                            | Each DB needs its own audit log; deployments may run across heterogeneous DBs |
| Compatibility window                 | Customers may skip up to N-2 minor versions; major-version transitions require running each major in sequence | Bounded matrix to test against                                                |
| Upgrade orchestration                | Single `platform upgrade` CLI; idempotent; safe to re-run after partial failure                               | One mental model for operators                                                |
| Rollback support                     | One step back via the orchestrator; older rollbacks require restoring from backup                             | Bounds the rollback test matrix                                               |
| HA upgrade strategy                  | Rolling restart, instance-by-instance; N and N-1 must coexist for one minor                                   | Customers run 2+ instances; can't downtime them simultaneously                |
| Upgrade order across DBs             | All schema migrations apply per-DB in parallel; version recording happens after all succeed                   | Fast-fail before any version row is written                                   |
| Health-gate post-upgrade             | All adapter `healthcheck` ports must return OK; Obj 3 SLO probes must be green within 60 seconds              | Don't claim success on a sick deployment                                      |
| Pre-flight backup gate               | Orchestrator refuses unless a recent backup exists per Obj 4's backup runbook                                 | Backups are the rollback floor                                                |
| Operator UI                          | Installation admin only; not workspace admin                                                                  | Upgrade is an installation-wide operation                                     |
| Telemetry                            | Standard structured events to Obj 3 stack; `platform.upgrade.*` event family                                  | Same observability path as everything else                                    |
| Audit                                | Every upgrade is an audit event at `info` level (success) or `warn` (failure / rollback)                      | Obj 7 audit conventions                                                       |

## 5. The Hard Parts

### 5.1 Version detection on a fresh install vs. an upgrade

A fresh install has no rows in `platform_versions`. An upgrade has at least one. The orchestrator must:

- Treat zero rows as "first install at the current `package.json` version" — write one row at install time, not retroactively
- Treat one or more rows as "upgrade from latest row's version to current `package.json` version"
- Refuse if the latest row's version is _higher_ than the current `package.json` version (downgrade is not supported except via explicit rollback)

The first-install path is wired into the existing first-run experience (Obj 16's installation onboarding).

### 5.2 Multiple databases, partial failure

Customers may run on Postgres only, or Postgres + MSSQL, or Postgres + Mongo, or all three. Migrations apply per-DB in parallel. If Postgres succeeds and Mongo fails:

- The orchestrator does _not_ write `platform_versions` rows on the successful DBs
- Recovery: operator re-runs `platform upgrade`; Postgres's already-applied migrations are no-ops via the existing `__platform_migrations` checksum table; Mongo retries
- If the failure is genuinely unrecoverable, operator runs `platform upgrade --rollback` to revert Postgres; this is bounded because no version row was committed

This is why version recording happens _after_ all migrations succeed, not interleaved.

### 5.3 Compatibility window enforcement

Skipping more than N-2 minor versions is unsafe because the migration set hasn't been tested as a single squashed apply. The orchestrator reads the latest `platform_versions` row and refuses if:

- Target major > current major + 1
- Target minor > current minor + 2 within the same major

Operators who insist on a longer skip must upgrade to an intermediate version first. The CLI surfaces the next eligible version explicitly.

### 5.4 HA version-skew

A two-instance deployment cannot be downtimed. Rolling upgrade:

1. Operator drains instance B (load balancer steers traffic to A)
2. Operator runs `platform upgrade` on instance B's connection, which applies migrations and records the new version
3. Instance B starts on the new version; A is still on the old version
4. Both must function with the new schema. **This is the contract:** every minor version's schema must remain compatible with the previous minor's code. We accept this constraint by:
   - Using only additive schema changes within a minor (no DROP COLUMN; no NOT NULL on existing nullable columns; no enum value removal)
   - Breaking changes scheduled to a major version with explicit downtime
5. Operator drains A and restarts on the new version; rolling complete

The orchestrator refuses to apply a migration tagged `breaking: true` if HA mode is enabled. Operators must explicitly opt in via `--allow-breaking` (which implies a maintenance window).

### 5.5 Rollback semantics

Rollback walks one step back: the latest `platform_versions` row is removed and the corresponding migration's `down` script applies on each DB. Constraints:

- Rollback only works if every active DB has a `down` migration for the most recent step. MSSQL adapter currently has up-only migrations (Obj 4a DoD didn't require down); rollback on MSSQL is **best-effort** and surfaces a warning. This will tighten when MSSQL down-migrations are added (tracked as a pre-customer item)
- Rollback further than one step requires backup restore — explicitly out of scope to keep the test matrix bounded

### 5.6 First install vs. upgrade ergonomics

The CLI is the same command. `platform upgrade` on a fresh DB writes the initial `platform_versions` row at the current code version after the standard migrations run. `platform upgrade --to <version>` is the explicit upgrade form for an existing install.

### 5.7 Pre-flight: backup verification

The orchestrator refuses if it cannot detect a recent backup. "Recent" is configurable but defaults to 24 hours. Detection:

- Postgres: queries the existing backup metadata Obj 4 records
- MSSQL: same
- Mongo: same

If the customer is running with backups disabled (not recommended, but allowed), `--skip-backup-check` is required and emits a warning audit event.

## 6. Component Specifications

### 6.1 PlatformVersionPort

New port at `packages/ports/platform-version/`:

```typescript
export interface PlatformVersion {
  releaseVersion: string; // semver
  appliedAt: Date;
  appliedBy?: UserId;
  schemaMigrationHighWater?: string;
  notes?: string;
}

export interface PlatformVersionPort {
  current(): Result<PlatformVersion | null, AppError>; // null on fresh install
  history(): Result<PlatformVersion[], AppError>;
  record(version: Omit<PlatformVersion, 'appliedAt'>): Result<void, AppError>;
  rollback(): Result<PlatformVersion, AppError>; // returns the removed version
}
```

One port; three adapters (`platform-version-postgres`, `platform-version-mssql`, `platform-version-mongo`). Each reads/writes its database's `platform_versions` table (PG/MSSQL) or collection (Mongo).

### 6.2 UpgradeOrchestrator

In `packages/core/src/upgrade/`:

- Reads current version per DB (via PlatformVersionPort)
- Determines target version (CLI arg or `package.json`)
- Runs compatibility checks (Section 5.3)
- Runs pre-flight checks (Section 5.7)
- Calls `SchemaMigrationPort.apply()` per active DB in parallel
- On all-success: calls `PlatformVersionPort.record()` per DB
- On any failure: surfaces detailed error, leaves state safely re-runnable
- Emits structured events at every step

### 6.3 CLI

New CLI at `apps/platform-cli/`:

- `platform version` — prints current code version and per-DB recorded versions
- `platform upgrade [--to <version>] [--dry-run] [--allow-breaking] [--skip-backup-check]`
- `platform upgrade --rollback` — one step back
- `platform upgrade --status` — last upgrade attempt status, including failures

Read-only commands run with the app user; mutating commands require the migrate user.

### 6.4 Operator UI panel

In the existing installation admin UI (Obj 16):

- Current version + per-DB versions
- "Check for upgrades" button (compares against a static eligibility map; no phone-home unless explicitly opted in)
- "Upgrade now" with a confirmation dialog showing: target version, downtime estimate, rollback availability, backup status
- Upgrade progress streamed via realtime (Obj 14)
- Audit log of past upgrades

### 6.5 Compatibility metadata in releases

Each minor release ships a `release-manifest.json` declaring:

```json
{
  "version": "1.2.0",
  "minPreviousVersion": "1.0.0",
  "breakingMigrations": [],
  "expectedDowntimeSeconds": 30,
  "rollbackSupported": true
}
```

The orchestrator reads this for the target version (bundled in the release) and uses it for the compatibility check.

### 6.6 Audit Events

```
platform.upgrade.started     {fromVersion, toVersion, dbs}
platform.upgrade.preflight   {checks, results}
platform.upgrade.migrated    {db, migrationsApplied, durationMs}
platform.upgrade.recorded    {db, version}
platform.upgrade.completed   {fromVersion, toVersion, totalDurationMs}
platform.upgrade.failed      {step, error, recoverable}
platform.upgrade.rolledback  {fromVersion, toVersion, reason}
```

### 6.7 Telemetry

Standard Obj 3 metrics:

- `platform_upgrade_duration_seconds{from,to,db}` (histogram)
- `platform_upgrade_failures_total{step,db}` (counter)
- `platform_version_info{version,db}` (gauge — exposed for Grafana version-tracking dashboards)

### 6.8 Operational Runbooks

- `platform-upgrade-procedure.md` — step-by-step for operators (single instance + HA)
- `platform-upgrade-rollback.md` — when and how to roll back; when to escalate to backup restore
- `platform-upgrade-stuck.md` — what to do when the orchestrator hangs or fails partway
- `platform-upgrade-version-skew.md` — HA rolling upgrade with both versions live

## 9. Verification Steps

1. **First-install records a version row.** Fresh DB; run `platform upgrade`; verify `platform_versions` has exactly one row matching `package.json` version.
2. **Same-version idempotent.** Run `platform upgrade` again on a DB already at the current version; no new row; no migrations applied; exit 0.
3. **Minor upgrade.** Boot platform at v1.0.0; upgrade to v1.1.0 build; verify schema migrations applied, version row written on every active DB, audit events emitted.
4. **Skip within window.** Boot at v1.0.0; upgrade to v1.2.0; succeeds.
5. **Skip too far.** Boot at v1.0.0; attempt upgrade to v1.3.0 (beyond N-2); orchestrator refuses with clear error; suggests v1.2.0 as intermediate.
6. **Major version refuses skip.** Boot at v1.5.0; attempt upgrade to v3.0.0; refused.
7. **Pre-flight backup gate.** No recent backup; orchestrator refuses unless `--skip-backup-check`; with the flag, emits audit warning.
8. **Pre-flight free disk gate.** Synthetically constrained free disk; orchestrator refuses.
9. **Partial failure recovery.** Multi-DB deployment; force Mongo migration to fail mid-apply; orchestrator does not write any version row; re-running succeeds.
10. **Rollback one step.** Upgrade to v1.1.0; run `--rollback`; version row removed; down migrations applied per DB; final state matches pre-upgrade.
11. **Rollback when MSSQL has no down.** MSSQL adapter emits warning; PG and Mongo roll back; operator sees clear status.
12. **HA rolling upgrade.** Two instances on v1.0.0; drain B, upgrade B to v1.1.0; A and B both serve traffic; drain A, restart A on v1.1.0; never down.
13. **HA breaking-migration refuses.** Release with `breaking: true`; orchestrator refuses without `--allow-breaking`.
14. **CLI `version` output.** Prints code version + per-DB recorded versions; matches DB rows.
15. **CLI `upgrade --status`.** After a failed run, surfaces last error with the failing step.
16. **Operator UI shows current state.** Installation admin sees version, history, eligibility.
17. **Operator UI streams progress.** Initiated upgrade streams events to the panel via realtime.
18. **Audit events present.** All events from Section 6.6 emitted with correct fields.
19. **Telemetry metrics present.** Prometheus scrape includes `platform_version_info` and `platform_upgrade_duration_seconds`.
20. **Cross-DB matrix.** PG-only, PG+MSSQL, PG+Mongo, all-three configurations all upgrade cleanly.
21. **Permissions.** Non-installation-admin cannot run `platform upgrade` via the UI; CLI requires the migrate user for mutating commands.
22. **Release manifest validation.** Orchestrator refuses an upgrade if the target version's release manifest is malformed or missing.

## 11. Anti-Patterns to Refuse

- **Phone-home upgrade checks.** The platform does not call out to a vendor-controlled service to check for upgrades. Self-hosted means self-discovery via release manifest. Opt-in telemetry is a separate, configurable feature.
- **Silent schema downgrade.** Downgrade is rollback-one-step only. Anything further requires backup restore with explicit operator action.
- **Skipping `platform_versions` writes on success.** Without the row, the next upgrade can't tell what version this is — the audit trail is the source of truth.
- **Letting one DB succeed and another fail without surfacing.** All-or-nothing on version recording. Operator sees the partial state explicitly.
- **Auto-upgrade.** No automatic upgrades. The operator triggers every transition.
- **Upgrade orchestrator hidden from audit.** Every step is audited. No "quiet" code paths.
- **Tying upgrades to platform release cadence.** Customers upgrade on their own schedule within the compatibility window. Don't force a customer who's on v1.0 to be off it within N weeks.

## 12. Open Questions for Confirmation Before Starting

1. **Compatibility window** — N-2 minor proposed. If customers commonly skip three or more minors, lengthen this; tradeoff is more migration paths to test.
2. **Release manifest hosting** — proposed bundled in the release artifact. Alternative is a separate signed JSON over HTTPS. Bundled is simpler; HTTPS allows post-release patches.
3. **Phone-home toggle** — proposed off by default. If we ever want telemetry on installed versions for security advisory targeting, the toggle exists; default stays off.
4. **MSSQL down migrations** — currently up-only; rollback on MSSQL is best-effort. Should this objective tighten that requirement before MSSQL goes to a paying customer? Recommendation: yes, file a follow-up amendment to Obj 4a.
