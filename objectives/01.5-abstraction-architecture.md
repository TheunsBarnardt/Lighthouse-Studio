# Objective 1.5: Abstraction Architecture

**Status:** Ready for development
**Prerequisites:** Objective 1 (repository and tooling) must be complete
**Blocks:** Every objective from this point forward

---

## 1. Purpose

Define the abstraction interfaces that make the platform genuinely portable across databases, runtimes, identity providers, storage backends, and infrastructure. These interfaces are the contracts between the platform's domain logic and the pluggable infrastructure beneath it.

Every persistence operation, every authentication call, every file write, every email, every published event, every vector search, every schema introspection — must go through one of these interfaces. No exceptions, no shortcuts, no "I'll abstract it later."

This objective produces no user-visible features. It produces the **contracts** that make the rest of the platform possible. A platform that can run on PostgreSQL, MSSQL, or MongoDB; on Linux or Windows; with built-in auth or enterprise SSO; on local disk, S3, or Azure Blob — that platform exists only because these contracts exist and are honored.

**This is the most important document in the foundation.** Every later objective references it. Get it wrong, and every subsequent objective compounds the error.

## 2. Scope

### In Scope

- Interface definitions for every pluggable subsystem
- Type contracts (input, output, error) for every interface
- Package structure for interfaces vs. implementations
- Discovery and registration patterns (how implementations are wired in)
- Capability negotiation (how the platform queries what an implementation supports)
- Standard error types crossing abstraction boundaries
- Testing patterns for both interface conformance and implementation correctness
- Documentation conventions for each abstraction
- ADRs documenting each abstraction's design

### Out of Scope (Belongs to Later Objectives)

- Any implementation of any interface (Objective 4 family for databases, future objectives for the rest)
- The Data Management Module's UI and APIs (separate objective)
- Schema designer, DDL generator, auto-generated APIs (separate objectives that build on top of these abstractions)
- AI Gateway interfaces (separate objective; the AI gateway has its own abstraction concerns)

## 2.1 Tenancy Model (Foundational, Read First)

The platform is **single-tenant at the installation level, multi-tenant within the installation**.

One organization installs the platform on their own infrastructure. That installation belongs entirely to them. There is no concept of multiple unrelated organizations sharing one running platform — each organization runs their own copy.

Inside a single installation:

- The **installation itself is the implicit organization.** No `Organization` entity in the data model.
- **Workspaces** are the top-level tenancy boundary. They subdivide the installation by department, product, client (if it's an agency), or any other grouping the customer chooses.
- **Projects** live inside workspaces.
- **Users** belong to the installation and may be members of zero, one, or many workspaces.
- **Cross-workspace queries are forbidden by default.** Authorization is enforced at the workspace boundary; nothing leaks between workspaces unless an explicit installation-admin operation crosses the line.

Implications for the abstractions in this document:

- Audit entries scope to `workspaceId`, never `organizationId`.
- The `UserDirectoryPort` manages the installation's users; users are global within the installation, but their _project-level access_ is mediated through workspace membership.
- Every multi-tenant-style query in the data layer carries a `workspaceId` filter, enforced at the service layer (and at the database layer where the database supports it).
- Backups, exports, and disaster recovery operate at the **installation** level. There is no per-tenant backup because there is one tenant.

If a future customer needs hosted multi-tenancy (one running instance, many isolated organizations), that is a different deployment mode and a different product configuration. It is explicitly out of scope.

## 3. Locked Decisions

| Decision                | Choice                                                                  | Rationale                                                                             |
| ----------------------- | ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------- |
| Tenancy                 | Single-tenant installation, workspaces inside                           | Matches self-hosted reality; simplest correct model                                   |
| Pattern                 | Hexagonal (ports & adapters) architecture                               | Separates domain from infrastructure cleanly; standard pattern for portable codebases |
| Interface location      | `packages/ports/*` (the "ports")                                        | Ports are the platform's view of what it needs                                        |
| Implementation location | `packages/adapters/*` (the "adapters")                                  | Adapters bridge ports to specific technologies                                        |
| Naming convention       | `*Port` for interfaces, `*Adapter` for implementations                  | Disambiguates from generic "service" or "provider" naming                             |
| Async style             | All I/O returns `Promise<Result<T, E>>`, never throws across boundaries | Errors are values; explicit handling is enforced                                      |
| Error model             | Typed result wrappers using `neverthrow` library                        | Prevents unchecked exceptions; makes error paths visible                              |
| Validation              | All inputs validated with zod at the port boundary                      | Ports are trust boundaries; never accept unvalidated input                            |
| Capability model        | Each adapter declares supported capabilities via a static manifest      | Platform queries capabilities at startup; degrades gracefully                         |
| Discovery               | Adapters registered via dependency injection at composition root        | No service locator antipattern; explicit wiring                                       |
| Testing                 | Every port has a conformance test suite that all adapters must pass     | Catches dialect drift across databases                                                |
| Documentation           | Every port has an interface contract document in `docs/contracts/`      | The interface contract is the spec                                                    |

## 4. The Abstractions

These are the ports the platform defines. Each one becomes an interface package, has at least one adapter package implementing it, and has a conformance test suite.

### 4.1 Persistence Ports

The largest and most complex set of abstractions. These are what make database portability real.

- **`SchemaIntrospectionPort`** — read the structure of the underlying database
- **`SchemaMigrationPort`** — apply, list, and revert migrations
- **`RepositoryPort<T>`** — generic CRUD with filtering, pagination, and querying for an entity type
- **`UnitOfWorkPort`** — transaction boundary for multi-repository operations
- **`QueryPort`** — typed query interface for read-only complex queries
- **`SchemaDdlPort`** — generate DDL statements for the target database from a normalized schema model

### 4.2 Identity Ports

- **`IdentityProviderPort`** — authenticate users, return verified identity claims
- **`UserDirectoryPort`** — manage user records (the platform's own user table, distinct from external identity)
- **`SessionPort`** — create, read, refresh, revoke sessions
- **`MfaPort`** — challenge and verify multi-factor authentication

### 4.3 Storage Ports

- **`ObjectStoragePort`** — read, write, delete, list, signed URLs for binary objects
- **`StorageMetadataPort`** — file metadata, ownership, permissions stored in the platform's database

### 4.4 Communication Ports

- **`EmailPort`** — send transactional emails
- **`SmsPort`** — send SMS messages (deferred implementation, but interface exists)
- **`PushNotificationPort`** — send push notifications (deferred implementation, but interface exists)

### 4.5 Eventing Ports

- **`EventBusPort`** — publish events; subscribe to events
- **`ChangeStreamPort`** — subscribe to data change events from the underlying database (for real-time features)

### 4.6 Search & AI Ports

- **`VectorStorePort`** — store and query vector embeddings
- **`FullTextSearchPort`** — full-text search across the platform's content
- **`EmbeddingPort`** — generate vector embeddings from text
- **`AiGenerationPort`** — generate text completions; this is the abstraction the AI worker uses

### 4.7 Background Work Ports

- **`JobQueuePort`** — enqueue, claim, complete, fail, retry background jobs
- **`SchedulerPort`** — register and trigger scheduled (cron-like) jobs

### 4.8 Observability Ports

These are defined here but implemented as part of Objective 3 (Observability Foundation). Listed here so the abstraction landscape is complete.

- **`LoggerPort`** — structured logging
- **`MetricsPort`** — counters, gauges, histograms
- **`TracerPort`** — distributed tracing spans

### 4.9 Audit Port

- **`AuditPort`** — append-only audit log writes; queryable for reads

### 4.10 Configuration Ports

- **`SecretStorePort`** — read secrets at runtime from a backing store
- **`FeatureFlagPort`** — query feature flag evaluations

## 6. Interface Contracts

This section defines the most critical ports in detail. The full set is documented in `docs/contracts/` once written; here I specify the foundational ones that every other objective will lean on.

### 6.1 RepositoryPort

The single most important interface in the platform. Defines what every entity persistence operation looks like.

```typescript
// packages/ports/persistence/src/repository.port.ts
import { Result } from 'neverthrow';
import { Filter, Sort, Page, PaginatedResult, EntityNotFoundError, ConflictError, PersistenceError, ValidationError } from './types';

export interface RepositoryPort<TEntity, TId = string> {
  /** Find an entity by id. Returns null if not found. */
  findById(id: TId): Promise<Result<TEntity | null, PersistenceError>>;

  /** Find one entity matching a filter. Returns null if not found. */
  findOne(filter: Filter<TEntity>): Promise<Result<TEntity | null, PersistenceError>>;

  /** Find many entities matching a filter, with paging and sorting. */
  findMany(opts: { filter?: Filter<TEntity>; sort?: Sort<TEntity>; page?: Page }): Promise<Result<PaginatedResult<TEntity>, PersistenceError>>;

  /** Count entities matching a filter. */
  count(filter?: Filter<TEntity>): Promise<Result<number, PersistenceError>>;

  /** Create a new entity. Throws ConflictError if id collision. */
  create(entity: TEntity): Promise<Result<TEntity, PersistenceError | ConflictError>>;

  /** Update an entity by id. Returns updated entity. */
  update(
    id: TId,
    changes: Partial<TEntity>,
    opts?: { expectedVersion?: number }, // optimistic locking
  ): Promise<Result<TEntity, PersistenceError | EntityNotFoundError | ConflictError>>;

  /** Soft-delete an entity (sets archived_at). Hard-delete is a separate method. */
  archive(id: TId): Promise<Result<void, PersistenceError | EntityNotFoundError>>;

  /** Hard-delete. Forbidden for most entities; allowed only via explicit policy. */
  hardDelete(id: TId): Promise<Result<void, PersistenceError | EntityNotFoundError>>;
}
```

Key points:

- **Result-based errors.** No throws across the boundary. `neverthrow`'s `Result<T, E>` type forces the caller to handle the error case at the type level.
- **Soft delete by default.** `archive` is the standard. `hardDelete` exists for cleanup workflows but is gated.
- **Optimistic locking baked in.** Every entity has an implicit version field; updates can require a specific version. Adapters enforce this.
- **No raw SQL leakage.** The `Filter<TEntity>` type is a structured object, translated by each adapter into the dialect's native filter syntax.

### 6.2 Filter<TEntity>

The query filter type that every database adapter must translate. Has to be expressive enough to cover real-world needs without being so expressive it leaks dialect.

```typescript
export type Filter<T> = { _and: Filter<T>[] } | { _or: Filter<T>[] } | { _not: Filter<T> } | FieldFilter<T>;

export type FieldFilter<T> = {
  [K in keyof T]?: T[K] | FieldOperator<T[K]>;
};

export type FieldOperator<V> =
  | { _eq: V }
  | { _neq: V }
  | { _in: V[] }
  | { _nin: V[] }
  | { _lt: V }
  | { _lte: V }
  | { _gt: V }
  | { _gte: V }
  | { _contains: V } // string contains, case-sensitive
  | { _icontains: V } // string contains, case-insensitive
  | { _starts_with: V }
  | { _ends_with: V }
  | { _is_null: boolean };
```

Each adapter implements a translator from this AST to its native query language:

- Postgres: parameterized SQL with `WHERE`
- MSSQL: parameterized SQL with `WHERE`
- Mongo: a Mongo query object

Not all operators may apply to all field types (e.g., `_starts_with` only on strings). Adapters return a typed error if asked to do something nonsensical.

Operators NOT in this set: anything dialect-specific. No regex, no JSON path queries, no full-text search. Those go through different ports.

### 6.3 SchemaIntrospectionPort

The engine that powers the visual schema designer in the data management module.

```typescript
export interface SchemaIntrospectionPort {
  listSchemas(): Promise<Result<SchemaInfo[], PersistenceError>>;
  listTables(schema?: string): Promise<Result<TableInfo[], PersistenceError>>;
  describeTable(schema: string, table: string): Promise<Result<TableDefinition, PersistenceError>>;
  listIndexes(schema: string, table: string): Promise<Result<IndexInfo[], PersistenceError>>;
  listForeignKeys(schema: string, table: string): Promise<Result<ForeignKeyInfo[], PersistenceError>>;
  listConstraints(schema: string, table: string): Promise<Result<ConstraintInfo[], PersistenceError>>;

  /** Capability check: does this database support the given feature? */
  supports(feature: SchemaFeature): boolean;
}

export type SchemaFeature =
  | 'schemas' // namespaced tables (Postgres yes, MSSQL yes, Mongo no)
  | 'foreign_keys' // (Postgres yes, MSSQL yes, Mongo no)
  | 'check_constraints'
  | 'computed_columns'
  | 'json_columns' // (Postgres jsonb, MSSQL json, Mongo native)
  | 'array_columns' // (Postgres yes, MSSQL no, Mongo yes)
  | 'partial_indexes'
  | 'unique_indexes'
  | 'spatial_indexes'
  | 'transactions'
  | 'change_streams';
```

The `TableDefinition` returned is a normalized model with columns, types (mapped to a platform-wide type system), nullability, defaults, and constraints. The platform's UI works against this normalized model; adapters translate to and from native.

### 6.4 SchemaDdlPort

The companion to introspection. Generates DDL for the target database from the normalized model.

```typescript
export interface SchemaDdlPort {
  /** Generate the DDL statements to create a new table. */
  createTable(definition: TableDefinition): Result<DdlStatement[], DdlError>;

  /** Generate DDL to alter a table from one definition to another. */
  alterTable(from: TableDefinition, to: TableDefinition): Result<DdlStatement[], DdlError>;

  /** Generate DDL to drop a table. */
  dropTable(schema: string, table: string, opts?: { ifExists?: boolean }): Result<DdlStatement[], DdlError>;

  /** Generate DDL for indexes, foreign keys, etc. */
  createIndex(index: IndexDefinition): Result<DdlStatement[], DdlError>;
  // ... etc

  /** Validate that a definition is expressible in the target dialect. */
  validate(definition: TableDefinition): Result<void, DdlError>;
}
```

The platform's normalized type system has a finite set of types: `string(n)`, `text`, `integer`, `bigint`, `decimal(p,s)`, `boolean`, `date`, `timestamp`, `timestamp_tz`, `uuid`, `binary`, `json`, and `array<T>` where T is one of the above.

Each adapter maps these to native types: Postgres `varchar`, `text`, `int`, `bigint`, `numeric`, `boolean`, `date`, `timestamp`, `timestamptz`, `uuid`, `bytea`, `jsonb`, `T[]`. MSSQL has different mappings (no native `uuid` — use `uniqueidentifier`; no native `array` — must error). Mongo has no DDL at all — `createTable` becomes "ensure collection exists with optional schema validator."

`validate()` lets the platform catch "you can't have an array column on MSSQL" before the user clicks save.

### 6.5 IdentityProviderPort

How the platform authenticates users. The "built-in auth" is an adapter implementing this; Entra ID is another adapter.

```typescript
export interface IdentityProviderPort {
  /** Begin a sign-in flow. May return a redirect URL (for OAuth/SAML) or a challenge (for password). */
  beginSignIn(input: SignInInput): Promise<Result<SignInChallenge, IdentityError>>;

  /** Complete a sign-in flow given the challenge response (password, code, OAuth callback). */
  completeSignIn(input: SignInCompletion): Promise<Result<VerifiedIdentity, IdentityError>>;

  /** Verify an existing token (session token, JWT, etc.) and return the identity. */
  verifyToken(token: string): Promise<Result<VerifiedIdentity, IdentityError>>;

  /** Sign out — invalidate the session. */
  signOut(token: string): Promise<Result<void, IdentityError>>;

  /** Capability check. */
  supports(feature: IdentityFeature): boolean;
}

export type IdentityFeature = 'password' | 'magic_link' | 'oauth' | 'oidc' | 'saml' | 'mfa_totp' | 'mfa_webauthn' | 'mfa_sms' | 'self_service_signup' | 'self_service_password_reset';

export interface VerifiedIdentity {
  subject: string; // the IdP's user identifier
  email?: string;
  emailVerified: boolean;
  displayName?: string;
  claims: Record<string, unknown>; // raw claims from the IdP
  providerId: string; // which adapter produced this (e.g., 'builtin', 'entra')
}
```

Notice: this port does NOT define users in the platform's database. That's the `UserDirectoryPort`'s job. `IdentityProviderPort` only verifies who someone _is_ — the platform then looks up or creates a corresponding user record via `UserDirectoryPort`.

This separation is what allows the platform to have its own users table (for project membership, roles, audit trails) while delegating authentication to whichever IdP the customer uses.

### 6.6 ObjectStoragePort

```typescript
export interface ObjectStoragePort {
  /** Upload an object. */
  put(key: string, data: Readable | Buffer, opts?: PutOptions): Promise<Result<ObjectInfo, StorageError>>;

  /** Download an object. */
  get(key: string): Promise<Result<Readable, StorageError | ObjectNotFoundError>>;

  /** Get metadata only. */
  head(key: string): Promise<Result<ObjectInfo | null, StorageError>>;

  /** Delete. */
  delete(key: string): Promise<Result<void, StorageError>>;

  /** List with prefix. */
  list(prefix: string, opts?: ListOptions): Promise<Result<ObjectInfo[], StorageError>>;

  /** Generate a signed URL for direct client access. May not be supported by all adapters. */
  signedUrl(key: string, method: 'GET' | 'PUT', opts: { expiresIn: number; contentType?: string }): Promise<Result<string, StorageError | NotSupportedError>>;

  supports(feature: StorageFeature): boolean;
}

export type StorageFeature = 'signed_urls' | 'multipart' | 'versioning' | 'object_locks' | 'public_read';
```

Local filesystem doesn't support signed URLs natively but can be wrapped in a server-side route that signs and serves. The capability flag lets the platform decide whether to expose direct uploads in the UI.

### 6.7 EventBusPort

```typescript
export interface EventBusPort {
  publish<T>(topic: string, event: T, opts?: PublishOptions): Promise<Result<void, EventError>>;

  subscribe<T>(topic: string, handler: EventHandler<T>, opts?: SubscribeOptions): Promise<Result<Subscription, EventError>>;
}

export interface EventHandler<T> {
  (event: T, context: EventContext): Promise<Result<void, unknown>>;
}

export interface Subscription {
  unsubscribe(): Promise<void>;
}
```

In-process adapter for development and single-instance deployments. Redis pub/sub or Postgres LISTEN/NOTIFY for production with multiple instances. The interface is identical.

### 6.8 ChangeStreamPort

The hardest interface to make uniform across databases, and the most valuable for the data management module's real-time feature.

```typescript
export interface ChangeStreamPort {
  /** Subscribe to changes on a specific table/collection. */
  watch(opts: {
    schema?: string;
    table: string;
    operations?: ChangeOperation[]; // ['insert', 'update', 'delete']
    filter?: Filter<unknown>; // server-side filter where supported
  }): AsyncIterable<ChangeEvent>;

  supports(feature: ChangeStreamFeature): boolean;
}

export type ChangeOperation = 'insert' | 'update' | 'delete' | 'truncate';
export type ChangeStreamFeature = 'before_after_image' | 'server_side_filter' | 'replay_from_position';

export interface ChangeEvent {
  table: string;
  schema?: string;
  operation: ChangeOperation;
  before?: Record<string, unknown>; // null for inserts; populated for updates/deletes if supported
  after?: Record<string, unknown>; // null for deletes; populated for inserts/updates
  occurredAt: Date;
  position: string; // opaque resume token
}
```

Adapters:

- **Postgres**: logical replication via `pgoutput` or wal2json, exposed as a stream
- **MSSQL**: Change Data Capture (CDC) tables polled or Service Broker messages
- **Mongo**: native change streams (the simplest of the three)

This abstraction is what powers "real-time updates in the UI when data changes" — the killer Supabase feature, now available on any database.

### 6.9 AuditPort

```typescript
export interface AuditPort {
  write(entry: AuditEntry): Promise<Result<void, AuditError>>;
  query(filter: AuditFilter, page: Page): Promise<Result<PaginatedResult<AuditEntry>, AuditError>>;
}

export interface AuditEntry {
  workspaceId: string;
  actorId: string | null; // null for system events
  action: string; // e.g., 'artifact.approved'
  resourceType: string;
  resourceId: string;
  before?: Record<string, unknown>;
  after?: Record<string, unknown>;
  metadata?: Record<string, unknown>;
  occurredAt: Date;
  ipAddress?: string;
  userAgent?: string;
}
```

The audit port is append-only by contract. Adapters enforce this at the storage level (no UPDATE or DELETE permissions on the audit table for the application user).

## 7. Composition Root

The `packages/composition` package is where adapters meet ports. It reads configuration and constructs the dependency graph.

```typescript
// packages/composition/src/compose.ts
import { env } from '@platform/config';
import { createPostgresAdapters } from '@platform/adapters-persistence-postgres';
import { createMssqlAdapters } from '@platform/adapters-persistence-mssql';
import { createMongoAdapters } from '@platform/adapters-persistence-mongo';
// ... etc

export interface PlatformContainer {
  persistence: {
    repositories: RepositoryRegistry;
    unitOfWork: UnitOfWorkPort;
    schemaIntrospection: SchemaIntrospectionPort;
    schemaDdl: SchemaDdlPort;
    schemaMigration: SchemaMigrationPort;
  };
  identity: IdentityProviderPort;
  userDirectory: UserDirectoryPort;
  storage: ObjectStoragePort;
  email: EmailPort;
  eventBus: EventBusPort;
  changeStream: ChangeStreamPort;
  search: FullTextSearchPort;
  vectorStore: VectorStorePort;
  embeddings: EmbeddingPort;
  ai: AiGenerationPort;
  jobs: JobQueuePort;
  scheduler: SchedulerPort;
  audit: AuditPort;
  logger: LoggerPort;
  metrics: MetricsPort;
  tracer: TracerPort;
  secrets: SecretStorePort;
  featureFlags: FeatureFlagPort;
}

export async function compose(): Promise<PlatformContainer> {
  const dbDriver = env.DATABASE_DRIVER; // 'postgres' | 'mssql' | 'mongo'
  const idDriver = env.IDENTITY_DRIVER; // 'builtin' | 'entra' | 'oidc' | 'saml'
  const storageDriver = env.STORAGE_DRIVER; // 'local' | 's3' | 'azure_blob'
  // ... etc

  const persistence = await selectPersistence(dbDriver);
  const identity = await selectIdentity(idDriver);
  const storage = await selectStorage(storageDriver);
  // ... etc

  return {
    persistence,
    identity,
    userDirectory: persistence.userDirectory,
    storage,
    // ...
  };
}

async function selectPersistence(driver: string) {
  switch (driver) {
    case 'postgres':
      return createPostgresAdapters({ url: env.DATABASE_URL });
    case 'mssql':
      return createMssqlAdapters({ url: env.DATABASE_URL });
    case 'mongo':
      return createMongoAdapters({ url: env.DATABASE_URL });
    default:
      throw new Error(`Unknown database driver: ${driver}`);
  }
}
```

The composition root is the **only** place in the platform where specific adapters are referenced. Every other piece of code receives a `PlatformContainer` (or a subset) via dependency injection and depends only on port interfaces.

This is what makes the platform testable: test code creates a `PlatformContainer` with in-memory adapters. It's what makes the platform portable: changing `DATABASE_DRIVER=mssql` switches the entire persistence layer.

## 8. Conformance Test Suites

Every port has a conformance test suite that all adapters must pass. This is the discipline that prevents "Postgres works, MSSQL has weird bugs" from happening silently.


## 9. Capability Negotiation

Some features cannot be supported on all databases. The platform handles this gracefully via the `supports()` method on each port and a centralized capability registry.

Example: the data management module's "real-time table view" feature requires `ChangeStreamPort.supports('server_side_filter')`. If the underlying adapter doesn't support that (say, because MSSQL CDC has limitations), the UI either:

- Disables the feature with an explanation ("Real-time filtering not supported on this database")
- Falls back to a less-efficient implementation (poll all changes, filter client-side)
- Offers an upgrade path ("Enable Service Broker to support this feature")

The platform never silently misbehaves. Capability mismatches are surfaced.

A central `CapabilityRegistry` aggregates all adapter capabilities at startup and is queryable from any code. UIs render conditionally on capabilities. Features that absolutely require a capability fail fast with a clear error if it's missing.

## 10. Error Model

Every port defines its own error union type. Errors cross the abstraction boundary as typed values, not exceptions.

```typescript
// packages/ports/persistence/src/errors.ts
export class PersistenceError extends Error {
  constructor(
    public code: PersistenceErrorCode,
    message: string,
    public cause?: unknown,
  ) {
    super(message);
    this.name = 'PersistenceError';
  }
}

export type PersistenceErrorCode = 'CONNECTION_FAILED' | 'TIMEOUT' | 'CONSTRAINT_VIOLATION' | 'DEADLOCK' | 'PERMISSION_DENIED' | 'UNKNOWN';

export class EntityNotFoundError extends Error {
  code = 'ENTITY_NOT_FOUND' as const;
  constructor(
    public entityType: string,
    public id: string,
  ) {
    super(`${entityType} not found: ${id}`);
  }
}

export class ConflictError extends Error {
  code = 'CONFLICT' as const;
  constructor(
    public reason: string,
    public details?: Record<string, unknown>,
  ) {
    super(reason);
  }
}
```

These are not thrown across the port boundary; they are returned in `Result<T, E>`. The `neverthrow` library's `Result` type forces the calling code to either handle the error or explicitly propagate it. Unhandled errors at the type level are caught by TypeScript.

Adapters catch driver-specific exceptions internally and translate them to typed errors:

```typescript
// in the postgres adapter
try {
  const row = await client.query('...');
  return ok(parseRow(row));
} catch (err) {
  if (isUniqueViolation(err)) return err(new ConflictError('Unique constraint violation'));
  if (isConnectionError(err)) return err(new PersistenceError('CONNECTION_FAILED', '...', err));
  return err(new PersistenceError('UNKNOWN', String(err), err));
}
```

This pattern is repeated in every adapter. It's verbose but it's what makes errors first-class.

## 11. Documentation Convention

Every port has a contract document at `docs/contracts/<port-name>.md`. The contract documents:

- The purpose of the port
- The methods, with semantic descriptions (not just type signatures)
- Pre-conditions and post-conditions for each method
- Invariants the adapter must maintain
- Performance expectations (e.g., "findById must be O(1)")
- Capability flags and what they mean
- Known divergences across adapters (e.g., "MSSQL adapter does not support `_starts_with` operator on `ntext` columns")
- Examples of correct usage
- Examples of common misuse and what happens

This document is the spec. If the contract document and the code disagree, the contract document wins and the code is fixed.

## 17. Anti-Patterns to Refuse

- **"I'll add the abstraction later."** No. Adding it later means changing every call site, which means you won't. Add it now.
- **Throwing exceptions across a port boundary.** No. Errors are values. The `Result` type is non-negotiable.
- **Importing an adapter directly from application code.** No. Application code receives the container; it does not reach into adapters.
- **Adding a method to a port to support one specific database's feature.** No. Either it's a capability flag with graceful degradation, or it doesn't belong in the port.
- **"This filter operator is too useful not to add."** No. The Filter AST is deliberately restrictive. If you need richer queries, that's the QueryPort, not RepositoryPort.
- **Skipping the conformance test for a new method.** No. New port method → conformance test in the same PR.
- **"MongoDB doesn't have transactions, so let's not require them."** No. The UnitOfWork port is required. Mongo's adapter implements it via session-scoped operations. The interface is the same.
- **Letting ports leak driver types.** A port file should never import `pg`, `mssql`, `mongodb`, `aws-sdk`, or any driver. If it does, the abstraction is broken.
- **Skipping the CLA "for now."** No. Once outside contributions land without a CLA, you've lost dual-licensing forever for those lines.
