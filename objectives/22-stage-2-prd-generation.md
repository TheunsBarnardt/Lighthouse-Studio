# Objective 22: Stage 2 — Requirements (PRD Generation)

**Status:** Ready for development
**Prerequisites:** Objective 20 (AI Pipeline Foundation), Objective 21 (Stage 1: Intent Capture) complete
**Blocks:** Objective 23 (Stage 3: Design Tokens), Objective 24 (Stage 4: Schema)

---

## 1. Purpose

Take an approved Intent Brief from Stage 1 and produce a structured **Product Requirements Document** (PRD) — the artifact that downstream stages (design, schema, code) consume to generate concrete deliverables. The PRD is the bridge between "what the user wants" (intent) and "what gets built" (everything after).

A good PRD is:

- **Specific enough** to guide design and engineering decisions
- **General enough** that a different competent team would arrive at similar implementation choices
- **Testable** — every requirement has an acceptance criterion that's verifiable
- **Bounded** — the PRD respects the intent's in-scope / out-of-scope boundaries
- **Reviewed** — humans approve sections; the AI doesn't ship requirements unilaterally

Done well, the PRD makes downstream stages possible. Done poorly, downstream stages either fail (because they can't infer enough) or hallucinate (because they fill gaps without authority). Stage 2 is the most critical AI stage in the pipeline because every later stage depends on its output.

This objective produces **the second AI artifact type** and the second user-facing AI feature. It introduces **section-level approval** as a pattern: a PRD is large enough that approving it whole-hog is impractical; section-by-section review is the discipline.

## 2. Scope

### In Scope

- **PRD artifact type**: structured schema for what a PRD contains
- **Section-based generation**: each section generated by a focused prompt
- **Section-level approval routing**: each section moves through approval independently; the PRD is "approved" when all sections are
- **Cross-section consistency checks**: the AI flags when sections contradict each other
- **Iterative refinement**: the user can request revisions of specific sections without regenerating everything
- **PRD viewer/editor UI**: a focused page for reviewing, editing, and approving a PRD
- **Acceptance criteria as first-class**: every functional requirement has explicit acceptance criteria
- **Traceability**: every PRD requirement traces back to a goal/criterion in the Intent Brief
- **Export formats**: markdown export for sharing with teams; PDF export for archival
- **Templates per project type**: starter PRD structures for common project shapes (CRM, blog, dashboard, etc.)
- **Quality signals specific to PRD**: section acceptance rate, cross-stage rejection feedback
- ADRs

### Out of Scope (Belongs to Later Objectives)

- Design specifications beyond what a PRD typically includes — visual design lives in Stage 3
- Database schema — Stage 4
- API specifications — Stage 7 (Code Generation)
- Wireframes / mockups — Stage 3 (Design Tokens) at the abstract level; Stage 6 (UI Generation) at the concrete level
- Cost / effort estimation — deferred; a future enhancement could add this
- Multi-PRD comparison ("which approach should we take?") — deferred
- AI-generated user research / persona development beyond what was captured in intent — deferred
- Stakeholder commenting / annotation on PRD sections — first version uses approval comments only; rich annotation deferred

## 5. The Hard Parts

**5.1 Why per-section generation, not whole-document**

A single prompt generating an entire PRD is tempting but fails:

- The output is too long for reliable single-pass quality (modern context windows are large but quality degrades at length)
- The prompt becomes too complex to maintain or test
- A single bad section poisons the whole; regenerating wastes everything
- Reviewers approve sections at different cadences; whole-doc approval doesn't fit real workflows

The platform's approach: each section has its own prompt, its own test suite, its own approval state. The "PRD artifact" is a composite that links to its 10 sections. Sections are independent units of work.

**5.2 Section dependencies**

Some sections naturally depend on others:

- "Functional Requirements" depends on "User Stories" (which depend on "Target Users")
- "Non-Functional Requirements" depends on "Functional Requirements" (sized to functionality)
- "Risks & Mitigations" depends on most other sections (you can't list risks until you know what's at stake)

The platform handles this with explicit dependency declaration in each section's prompt configuration. The orchestrator runs sections in topological order; sections without dependencies run first; downstream sections receive their parents' outputs as input.

This means generating a full PRD takes some time (the slowest path through the dependency graph). The UI shows progress: "Section 4 of 10 generating..."

**5.3 Cross-section consistency**

Even with careful per-section prompts, contradictions creep in:

- Section 5 lists FR-12 "Real-time updates"; Section 6 says "performance acceptable up to 5-second latency"
- Section 3 lists a persona "Casual user"; Section 6 specifies "All users must have admin training"
- Section 8 lists "out of scope: mobile app"; Section 5 includes "FR-23: mobile push notifications"

A separate consistency-check prompt runs after all sections are drafted. It receives the full PRD and looks for contradictions. Found contradictions are surfaced to the user as warnings ("Section 5 and Section 8 may conflict on mobile scope; please review").

The user resolves contradictions either by editing one of the sections or by clicking "regenerate this section taking the conflict into account."

**5.4 Section-level approval routing**

The approval routing engine from Objective 6 already supports configurable routes per stage. For PRD, the routing is configured per _section type_:

```yaml
approval_routes:
  prd:
    overview:
      approvers: [business_analyst]
      mode: any_of
    functional_requirements:
      approvers: [business_analyst, architect]
      mode: all_of
    non_functional_requirements:
      approvers: [architect]
      mode: any_of
    # ...
```

In a solo workspace, all sections route to the workspace owner; one click per section. In an enterprise workspace, BA approves user-facing sections; Architect approves technical sections; both approve high-stakes ones. Same engine, different config.

The PRD is "approved" when all 10 sections are approved. The PRD viewer UI shows the current approval state per section: pending, approved by [user], rejected with [feedback].

**5.5 Iterative refinement without losing work**

Common scenario: the user approves 8 sections; section 9 they want to revise. The platform supports this without regenerating everything:

- Each section's `revise` action takes optional user feedback
- The revision prompt receives: the original intent brief + the current section + the user's feedback + the OTHER 8 approved sections (for consistency)
- The revised section becomes a new version; the old version is preserved
- The other 8 sections are untouched; their approvals stand

For larger structural changes (e.g., the user realizes the intent was wrong), the platform offers "regenerate full PRD" which re-runs all sections — but warns that approved sections will be invalidated.

**5.6 Traceability — every requirement traces back**

Each functional requirement has a `traces_to` field:

```typescript
{
  id: 'FR-12',
  title: 'Real-time data updates',
  description: '...',
  acceptance_criteria: [...],
  priority: 'must',
  traces_to: [
    { type: 'intent_brief', field: 'goals', goal_id: 'goal-3' },
    { type: 'intent_brief', field: 'target_users', persona_id: 'persona-1' },
  ],
}
```

The traceability is structural: every requirement points back to specific intent fields. This serves three purposes:

1. **Audit**: if a requirement seems off, the reviewer can trace back to the intent and verify
2. **Coverage**: the platform can detect intent goals that don't have any requirements addressing them ("you said real-time was a goal but no requirement mentions it")
3. **Downstream propagation**: when Stage 4 generates the schema, it traces back through requirements to intent; when Stage 6 generates UI components, it does the same. The full dependency chain is queryable

The traceability check is a dedicated prompt that runs after generation: it verifies every intent goal has at least one supporting requirement, and flags goals that don't.

**5.7 Acceptance criteria as testable specifications**

Every functional requirement has acceptance criteria in Gherkin-influenced format:

```typescript
{
  id: 'FR-12',
  title: 'Real-time data updates',
  acceptance_criteria: [
    {
      given: 'A user is viewing a record in the data browser',
      when: 'Another user updates the same record',
      then: 'The first user sees the updated values within 2 seconds without manual refresh',
    },
    {
      given: 'A user has lost network connectivity',
      when: 'They regain connectivity',
      then: 'Updates that occurred during the outage are reflected within 5 seconds',
    },
  ],
}
```

These criteria are inputs to Stage 8 (Test Generation) — the platform generates tests from the criteria. The criteria are also human-verifiable: a QA tester can manually check them without needing to read the generated tests.

For non-functional requirements, criteria are usually metric-based:

```typescript
{
  id: 'NFR-3',
  category: 'performance',
  description: 'API response times',
  acceptance_criteria: [
    { metric: 'list_endpoint_p95_latency', threshold: '< 200ms', measurement: 'across 1000 requests under load' },
    { metric: 'list_endpoint_p99_latency', threshold: '< 500ms' },
  ],
}
```

Both formats serve the same purpose: testable, verifiable, unambiguous.

**5.8 Templates per project type**

A PRD for a CRM differs from a PRD for a blog. The platform's templates seed different starting structures:

- **CRM template**: emphasizes contact management, deal stages, reporting; user stories include sales reps, managers, admins
- **Blog template**: emphasizes content, readers, comments, moderation
- **Internal tool template**: emphasizes employee personas, workflows, integration with existing systems
- **Customer portal template**: emphasizes customer-facing UX, account management, support flows

Templates aren't strict scaffolds — they're starter prompts. The PRD generation respects the user's intent regardless of template; the template just informs initial section emphasis.

For workspaces with their own conventions, custom templates can be defined.

**5.9 Editing UX: preserving the structure**

When the user edits a section, the platform tries to preserve the structure:

- Functional requirements: edits to a single FR; can add/remove FRs; renumbering happens automatically (FR-12 stays FR-12 even if FR-3 is removed; gap is ok)
- User stories: same pattern
- Non-functional requirements: same
- Free-form sections (Overview, etc.): freely editable as markdown

Structured edits use form-based UI where each item is a card; free-form sections use a markdown editor.

The user can switch between "structured" and "raw markdown" views. Edits in markdown mode are parsed back into structure; if parsing fails (the user broke the format), a clear error guides recovery.

**5.10 Re-running on intent changes**

If the upstream intent brief is modified after the PRD is generated, the PRD becomes "potentially out of date." The platform:

- Marks the PRD with a "stale" indicator
- Shows which intent fields changed since generation
- Offers "regenerate affected sections" — runs only the sections that depend on changed intent fields
- The user can approve the regenerated sections without re-approving unchanged sections

This feels right: incremental updates instead of throwing everything away. The user only re-reviews what actually changed.

**5.11 Quality signals specific to this stage**

Beyond Objective 20's generic signals:

- **Per-section acceptance rate**: which sections get approved on first pass vs. revised? Reveals which prompts need iteration
- **Cross-section contradiction rate**: how often does the consistency check find issues? High rate = orchestration needs work
- **Downstream rejection feedback**: when Stage 4 (Schema) rejects a PRD as insufficient, what was missing? Feedback loop into PRD prompts
- **Time per section**: how long does each section take to approve? Long times = friction in the prompt or the UI
- **Number of revisions per section before approval**: tracks prompt quality

These feed dashboards that the platform team uses to iterate prompts. Customer-facing dashboards show aggregated patterns ("functional requirements typically need 1.3 revisions before approval").

### 6.2 The PRD Artifact Model

```typescript
interface Prd {
  intentBriefId: string; // parent
  templateUsed?: string; // which starter template, if any
  sectionIds: Record<PrdSectionType, string>; // map of section type → section artifact id
  generationMetadata: PrdGenerationMetadata;
  consistencyReport?: ConsistencyReport;
  traceabilityReport?: TraceabilityReport;
  stalenessIndicators?: StalenessIndicator[];
}

type PrdSectionType = 'overview' | 'goals_and_success_metrics' | 'target_users_and_personas' | 'user_stories' | 'functional_requirements' | 'non_functional_requirements' | 'constraints_and_assumptions' | 'out_of_scope' | 'open_questions' | 'risks_and_mitigations';

interface PrdSection {
  id: string; // artifact ID (each section is its own artifact)
  prdId: string; // parent PRD artifact
  sectionType: PrdSectionType;
  status: ArtifactStatus;
  currentVersion: number;
  content: PrdSectionContent; // shape varies by section type
  reasoning: ReasoningRecord;
  approvalId?: string;
  qualitySignals?: QualitySignals;
  createdAt: Date;
  updatedAt: Date;
}

type PrdSectionContent = OverviewContent | GoalsAndMetricsContent | TargetUsersContent | UserStoriesContent | FunctionalRequirementsContent | NonFunctionalRequirementsContent | ConstraintsAndAssumptionsContent | OutOfScopeContent | OpenQuestionsContent | RisksAndMitigationsContent;

interface FunctionalRequirementsContent {
  requirements: FunctionalRequirement[];
}

interface FunctionalRequirement {
  id: string; // FR-1, FR-2, etc.
  title: string;
  description: string;
  priority: 'must' | 'should' | 'could' | 'wont';
  acceptanceCriteria: AcceptanceCriterion[];
  tracesTo: TraceabilityRef[];
}

interface AcceptanceCriterion {
  given: string;
  when: string;
  then: string;
}

interface TraceabilityRef {
  type: 'intent_brief' | 'prd_section' | 'requirement';
  artifactId: string;
  fieldPath: string; // e.g., "goals.goal-3", "user_stories.US-12"
}
```

Each section is a separate artifact (per Objective 20). The PRD artifact is a composite that references all 10 sections.

## 9. Verification Steps

1. **Generate a PRD** from an approved intent brief; all 13 sections produced; PRD artifact references them.

2. **Section dependency order respected**: independent sections run in parallel; dependent sections wait.

3. **Each section has reasoning**: reasoning visible in UI; explains why specific requirements emerged.

4. **Functional requirements have acceptance criteria**: every FR has at least one Given/When/Then; no FRs without criteria.

5. **Traceability**: every requirement traces back to specific intent fields.

6. **Traceability check**: runs successfully; flags intent goals without supporting requirements.

7. **Consistency check**: detects contradictions in test scenarios where they exist.

8. **Edit a section**: changes persist; new version created; other sections unchanged.

9. **Regenerate a section with feedback**: new version reflects feedback.

10. **Section approval (solo)**: workspace owner approves; section status transitions.

11. **Section approval (enterprise with all_of mode)**: requires all configured approvers.

12. **PRD considered complete**: when all 13 sections approved.

13. **Section rejection**: section returns to draft with feedback visible.

14. **Staleness detection**: modifying the intent brief flags affected sections; non-affected sections retain approval.

15. **Affected sections regeneration**: only the affected sections are regenerated.

16. **Export to markdown**: full PRD exports as well-formatted markdown; sections in correct order; traceability preserved.

17. **Templates**: selecting a CRM template biases initial generation toward CRM-typical sections; user can deviate.

18. **Cross-database**: same generation flow works on Postgres, MSSQL, Mongo workspaces.

19. **Audit events**: every lifecycle action emits expected entries.

20. **Cost tracking**: PRD generation cost recorded per section; total cost visible.

21. **Provider failover**: if primary provider fails mid-generation, fallback completes; sections from each provider attributed correctly in metadata.

22. **Output schema validation**: malformed section outputs trigger one retry; persistent failure surfaces error.

23. **PII redaction**: any PII referenced in the intent brief is redacted in PRD prompts; preserved in display.

24. **Performance**: full PRD generation completes within 5 minutes typical case.

25. **Quality signals**: per-section acceptance rates recorded; dashboards show patterns.

26. **Concurrent revisions**: two reviewers editing different sections simultaneously don't collide; same section conflicts via optimistic locking.

27. **Locked Decisions section** emits a table with Decision · Choice · Rationale columns; downstream stage prompts read this table as binding constraints.

28. **Architectural Overview section** emits a data-flow diagram (mermaid or ASCII box-and-arrow) plus a narrative; the diagram references components named in Section 6.

29. **The Hard Parts section** identifies at least one non-trivial design decision per substantial PRD; flagged decisions become inputs to Stage 4 (Schema) and Stage 7 (Code) prompts.

30. **Component Specifications section** emits typed contracts (interface stubs / data models) for each major component named in Section 4; Stage 7 consumes these as scaffolding.

31. **ADRs to Write section** emits 3–10 ADR titles with rationale stubs for the customer's app; Stage 7 scaffolds them into the generated app's `docs/adr/` folder per ADR-0261.

32. **Verification Steps section** emits numbered, testable end-to-end checks in the platform's own style; Stage 8 (Test Generation) consumes each step as a test seed.

33. **Definition of Done section** emits categorized checkboxes; each downstream stage (4, 6, 7, 8, 9) reads its slice as a completion gate.

34. **Anti-Patterns section** emits explicit "we don't do X" rules; downstream prompts treat them as negative constraints (refuse to generate features the PRD has refused).

35. **Implementation Order section** emits a numbered build sequence; Stage 7 (Code Generation) and Stage 9 (Deployment) respect dependency order.

36. **Cross-section trace from rigor sections back to Intent Brief**: every Locked Decision, every ADR-to-Write, every Hard Part traces back to a goal/constraint in the Intent Brief — same traceability discipline as Functional Requirements.

If all 36 pass, the objective is met.

## 11. Anti-Patterns to Refuse

- **Single-prompt full-PRD generation.** Per-section is the discipline; single-prompt fails at scale.
- **Skipping the consistency check.** Contradictions accumulate without it; downstream stages suffer.
- **Skipping the traceability check.** Coverage gaps mean intent goals get lost.
- **Letting sections approve without acceptance criteria.** Functional requirements without testable criteria are wishes, not requirements.
- **Whole-PRD approval.** Section-level granularity matches real review workflows.
- **Regenerating everything when one section needs revision.** Wastes work; user resists the friction.
- **Hardcoding the section list.** Locked but enumerable; downstream stages reference the enum.
- **Treating the rigor sections (Locked Decisions, Architectural Overview, Hard Parts, Component Specs, ADRs, Verification, DoD, Anti-Patterns) as optional.** They are the bridge between "requirements" and "buildable specification"; an app that ships without them inherits less rigor than the platform demands of itself.
- **Letting downstream stages bypass the PRD's Anti-Patterns.** A feature the PRD explicitly refused must not appear in the generated app. Stage 6/7/8 prompts treat Anti-Patterns as hard constraints.
- **Auto-progressing to Stage 3.** User explicitly transitions; respect the pace.
- **Ignoring staleness on intent changes.** Cascading rot; users discover problems later when they've moved on.
- **Templates as constraints.** Hint, not requirement; AI deviates when intent demands.
- **Missing reasoning in section content.** Reasoning is non-optional per Objective 20.

## 12. Open Questions for Confirmation Before Starting

1. **13-section count** — locked per ADR-0260; mirrors the platform's own objective document structure. Optionality complicates downstream consumption.

2. **PDF export** — proposing markdown primary, PDF deferred until a PDF generation service is built. Acceptable, or include PDF in v1?

3. **Acceptance criteria format** — proposing Gherkin-influenced (Given/When/Then). Some teams prefer simple bulleted criteria. Recommendation: Gherkin for functional; flexible for non-functional.

4. **Template count** — proposing 5–6. Worth more (10–15) for breadth, or fewer high-quality? Recommendation: ship with 5–6; add as customer demand reveals gaps.

5. **Approval routing per section type** — proposing per-section-type configurability. Some workspaces will want one config for the whole PRD. Recommendation: per-section-type with sensible defaults; one-config workspaces just set all section types to the same approvers.

6. **Section regeneration cost cap** — should regenerating a single section have a limit (e.g., $0.50 per regeneration)? Recommendation: yes; per-action cost ceilings prevent runaway iteration cost.

7. **Open Questions section behavior** — does it need approval like other sections? Or is it more of a "thinking out loud" space? Recommendation: same approval flow; the questions ARE outputs that downstream stages may use to ask the user later.
